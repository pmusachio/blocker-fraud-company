# Blocker Fraud Company

A data science project that detects fraudulent financial transactions, built on the
Kaggle [Synthetic Financial Datasets For Fraud Detection / PaySim](https://www.kaggle.com/datasets/ntnu-testimon/paysim1)
dataset. The project follows the end-to-end workflow described in *Hands-On Machine Learning*
(Aurelien Geron): frame the business problem, explore and prepare the data, build a
reproducible Scikit-Learn pipeline, evaluate it with business-relevant metrics, and ship
the score through an API, a dashboard and a chat bot.

## 1. Business Problem

A financial operation needs to block fraudulent transactions without slowing down
legitimate, high-value transfers.

**Goal:** detect fraudulent transactions while prioritizing recall, precision@k and the
financial value protected.

**Key metrics:** average precision, recall, precision@k and expected protected revenue.

## 2. Business Assumptions

- Fraud is rare, so plain accuracy does not represent business quality.
- False negatives cost more than the extra analyst time spent reviewing false positives.
- The model score should feed a prioritized investigation queue, not an automatic block.

## 3. Solution Strategy

The CRISP-DM-style steps below mirror the modeling chapters of the book:

1. **Data Description:** validate schema, dimensions, missing values, types and granularity.
2. **Feature Engineering:** create domain-driven variables (balance deltas, transaction cycle, ratios).
3. **Data Filtering:** drop columns that leak information or add no analytical value.
4. **Exploratory Data Analysis:** validate hypotheses and separate signal from noise.
5. **Data Preparation:** impute, scale and encode variables inside a single Scikit-Learn `Pipeline`.
6. **Feature Selection:** separate IDs, target, input features and discarded columns.
7. **Machine Learning Modelling:** train a reproducible baseline and evaluate technical metrics.
8. **Hyperparameter / Fit Strategy:** reserve room for tuning, threshold calibration and model comparison.
9. **Business Translation:** convert technical metrics into decisions, prioritization and protected revenue.
10. **Delivery:** publish the score through a REST API, a Streamlit dashboard and a Telegram bot.

## 6. Top Data Insights and Hypotheses

- `TRANSFER` and `CASH_OUT` transaction types concentrate most of the fraud risk.
- Inconsistent balance deltas (origin/destination) are strong fraud signals.
- High amounts must be evaluated together with origin and destination balances, not in isolation.

## 7. Model or Analysis Applied

Binary classification with `LogisticRegression` (`class_weight="balanced"`) inside a
`ColumnTransformer` + `Pipeline`, combining balance-delta features with ranking metrics
such as precision@k — the same pattern used for the imbalanced classification chapters
of the book.

## 8. Performance and Business Results

Running the main pipeline writes metrics and artifacts to `reports/` and `models/`:

- [`reports/metrics.json`](reports/metrics.json)
- `models/model.joblib`

**Business translation:** the score lets the fraud team estimate how much financial value
stays protected by reviewing the highest-scored transactions first (precision@k and lift@k).

## 9. Repository Structure

- `configs/project.toml`: project contract — data, target, metrics and modeling parameters.
- `src/blocker_fraud_company/`: modular Python package for data loading, features, modeling and serving.
- `notebooks/`: the analytical journey as Jupyter notebooks.
- `data/raw/`: raw data downloaded from Kaggle.
- `reports/`: metrics, profiles and results generated by the pipeline.
- `docs/deployment.md`: deployment and consumption notes.
- `app/`: Streamlit dashboard and Telegram webhook for consuming the score.
- `models/`: trained model artifact (`model.joblib`).

## 10. How to Run This Project (for recruiters and reviewers)

You only need Python 3.10+ and the PaySim CSV. Everything below runs from the repository root.

### Step 1 — Clone and install

```bash
git clone <REPOSITORY_URL> blocker-fraud-company
cd blocker-fraud-company
python -m venv .venv
source .venv/bin/activate          # Windows: .venv\Scripts\activate
make install                       # pip install -r requirements.txt
```

### Step 2 — Get the data

Download `transactions.csv` from the [PaySim Kaggle page](https://www.kaggle.com/datasets/ntnu-testimon/paysim1)
and save it as `data/raw/transactions.csv`. Or, with a Kaggle API token configured:

```bash
pip install kaggle
kaggle datasets download -d ntnu-testimon/paysim1 --unzip -p data/raw
mv "data/raw/PS_20174392719_1491204439457_log.csv" data/raw/transactions.csv
```

### Step 3 — Run the pipeline

```bash
make profile   # writes reports/data_profile.json
make train     # trains the model, writes models/model.joblib and reports/metrics.json
```

Inspect the results:

```bash
cat reports/metrics.json
```

### Step 4 — Try the score live

Open a second terminal for each consumption channel you want to try.

**REST API (FastAPI):**

```bash
make api                                    # serves http://127.0.0.1:8000
PYTHONPATH=src python scripts/sample_api_request.py
```

**Streamlit dashboard:**

```bash
python -m pip install -r requirements-app.txt
make app                                    # opens http://127.0.0.1:8501
```

Upload a CSV of transactions or paste a single transaction as JSON to get a fraud score.

**Telegram bot** (optional, requires the API running and a bot token):

```bash
export TELEGRAM_TOKEN="<your-telegram-bot-token>"
export FRAUD_API_URL="http://127.0.0.1:8000/predict"
python app/telegram_bot.py
```

### Step 5 — Run the tests

```bash
make test
```

See [`docs/deployment.md`](docs/deployment.md) for more details on each delivery channel.
